# Ideas & To-Dos (loose backlog)

Not commitments — just a dumping ground for bugs and concepts to revisit.

## Bugs

- **Lock/unlock animation direction** — the underline/lock animation stems from the left side of the ingredient, but the lock icon itself sits on the right. Animation origin should match the lock's position. (`underlineIn` animation, `src/components/v2/IngredientDisplay.tsx`)
- **Gray X on saturated backgrounds** — the remove (X) icon color looks off against saturated ingredient backgrounds; needs a better color/contrast treatment (`CustomUnlock` icon, `src/components/v2/IngredientDisplay.tsx:206`).
- **Taste Lab text shrinks at 3 slots** — going from 2 → 3 slots shrinks ingredient text even though it's really a width issue (2 slots stay half-width, 3rd slot goes full-width instead of all three sharing evenly). Text size shouldn't need to drop.
- **Taste Lab text squash/stretch** — adding/removing slots causes a distorted squash-and-stretch effect on the text during the transition; needs a cleaner transition.
- **Steering example dish mismatch** — when using category/tag steering in Classic mode (e.g. clicking "Drinks" or "Dessert"), the "seen in" example dish shown below doesn't always match the selected category (e.g. selecting Dessert showed a minestrone-type dish because ingredients overlapped). The canonical example needs real contextual awareness of the selected steering category, not just shared ingredients. Acknowledged as a bigger data project, but core to why the feature exists — a wrong example undermines trust in the steering itself.
  - Concrete repro: banana + raisin, category pill "sandwiches & burgers" selected, "seen in" showed *"How To Make Ice Cream Without An Ice Cream Maker"*. Category, ingredients, and cited recipe all disagree with each other — need a systematic way to keep the selected steering category, the generated combo, and the "seen in" recipe all consistent with each other, not three independently-plausible-but-mismatched signals.
- Remove **skyr** from the ingredient list (`src/data/ingredientProfiles.ts`).
- **Animate the ampersand on mobile** — the "&" between ingredients is static when adding/removing ingredients; it should get the same enter/exit animation the ingredients themselves use.
- **"Pasta" the ingredient vs. "pasta" the dish type** — confusing overlap in the UI, especially on the landing search, where it's unclear whether you're searching/selecting the ingredient or steering toward the dish category. Needs some disambiguation (labeling, grouping, icon, etc.).
- **Inconsistent result from landing search entry points** — adding a dish or cuisine on the landing page generates a full 4-ingredient pairing, but adding a starting ingredient just gives you that single ingredient with no pairing generated. These entry points should behave consistently.
- **Duplicate "seen in" dishes: singular vs. plural** — e.g. "seen in ‘California Roll’, ‘California Rolls’" shows as two separate dish entries when they're the same dish. Need to dedupe/normalize singular vs. plural dish names so they're treated as one. Same issue seen with "Biscochitos" / "Bizcochitos" (alternate spelling, not just plural) — dedupe logic needs to cover both plural and spelling variants.
- **Landing search matches mid-string instead of prefix** — typing "c" surfaces "Mexican" and "French" (which merely contain a "c") ahead of/alongside "Chinese" and "Cajun & Creole". Search should prioritize/require matches that start with the typed letters, not just contain them anywhere. Likely affects ingredient search too, not just cuisine/dish search. Consider also generally preferencing ingredient results over cuisine/dish results in the landing search.
- **Mobile: adding first ingredient from empty state leaves a blank second slot** — selecting an ingredient from search on mobile shows that ingredient plus an empty slot; tapping "+" is needed to fill it. Instead, selecting an ingredient from search should immediately generate a full 2-ingredient pairing by default, with the searched-for ingredient locked (so hitting Generate afterward still respects/includes it).
- **Landing search → Classic slot-count inconsistency** — coming off the landing search with ingredients (as opposed to cuisine/dish selection), the resulting combo sometimes shows a stray comma after the first ingredient in what's supposed to be a 2-ingredient pairing, and slot count/state seems inconsistently initialized. A 2-ingredient pairing should never render a comma. Doesn't seem to happen as often when starting from a cuisine or dish selection — suggests the ingredient-driven landing path isn't setting up slot state the same way.
- ~~**Missing ampersand after generating from the landing surface**~~ — FIXED (`seedFreshCombo` in `src/FlavorFinderV2.tsx` now resets `targetIngredientCount` to 2 in both branches, so a stale target left by a preset/tag no longer makes the fresh landing pair render with commas + no "&"). Repro was: fresh generate, back to landing, generate again → no "&"; separators only corrected on a subsequent Generate because they derive from `maxSlots`/`targetIngredientCount`, which the landing seed wasn't updating.
- **Favicon stretches tall on soft refresh** — favicon renders correctly square on a hard reload, but on a normal page refresh it gets stretched taller than it should be. Likely a caching/sizing quirk (maybe a stale cached favicon at the wrong aspect ratio, or CSS/layout affecting an inlined favicon) rather than the source asset itself.

## Concepts / ideas

- **Richer lock/hover interaction** — rework what happens on hover/click for an ingredient slot beyond just lock/remove. Consider surfacing a small hover menu per slot with Taste Lab-style role controls (taste/category constraints) directly in Classic mode.
- **Unify Classic and Taste Lab** — long-term goal: converge the two modes so slot/role settings (taste/category constraints) work the same way in both, rather than maintaining two separate mental models. Classic is the one that's gotten the most attention/love; Taste Lab feels newer and a bit clunkier by comparison — any unification should lean toward Classic's feel as the base.
- **Open question: does it make more sense to visualize taste or category?** Worth deciding which axis is the primary one to surface visually across the app (slots, steering, filters, etc.) — taste profile vs. ingredient category — rather than treating them as equally weighted.
- **Floating "intelligent add" UI instead of a bottom toolbar** — another Classic/Taste Lab convergence idea: replace (or supplement) the bottom toolbar with a floating UI trailing below the selected ingredients that suggests complementary additions — e.g. "add cheese" or "add sour" — based on a read of the current combo's taste profile. Since every ingredient already carries a taste score, this could resurface something like an early prototype mode that suggested complementary tastes for a pairing, so adding the next ingredient is guided rather than fully random.
- Taste Lab can be left alone for now and revisited once the above bugs are addressed.
